# Vibe Coding — AI 原生编程方法论

## 触发条件

用户输入涉及以下任一内容时激活：
- "vibe coding" / "vibe 编程" / "氛围编程"
- "用 AI 写代码但不想看代码"
- "Accept All" / "全盘接受 AI 生成的代码"
- "像 Karpathy 说的那样编程"
- "对话式编程" / "自然语言编程"
- 讨论 AI 编码哲学/方法论/范式转变
- 需要设计 AI 驱动的开发工作流

## 起源

Vibe Coding 由 **Andrej Karpathy** 于 2025 年 2 月 2 日提出：

> "一种全新的编程方式，你完全臣服于 vibe……你甚至忘记了代码的存在。我几乎不碰键盘。总是点'Accept All'，不看 diff。出错时直接把错误信息丢给 AI，不加注释，通常这就修好了。代码超出了我日常的理解范围……但这不算真正写代码——我只是看东西、说东西、运行东西、复制粘贴东西，而它基本都能用。"

## 七项核心原则

| # | 原则 | 说明 |
|---|------|------|
| 1 | **物质脱离 (Material Disengagement)** | 程序员刻意脱离对代码的直接操作（编写/阅读/编辑），信任 AI Agent 执行这些操作 |
| 2 | **结果导向验证 (Outcome-Oriented Validation)** | 通过观察行为和结果来验证代码，而非逐行理解 |
| 3 | **对话式开发 (Conversational Development)** | 开发通过自然语言对话进行；AI 解读模糊的、高层次意图 |
| 4 | **迭代目标满足 (Iterative Goal Satisfaction)** | 在提示 AI → 评估生成代码（快速扫描+测试）→ 手动编辑或继续提示 之间交替 |
| 5 | **动态信任 (Dynamic Trust)** | 对 AI 的信任不是全盘接受，而是通过迭代验证建立；开发者保持选择性和战略性监督 |
| 6 | **上下文工程 (Context Engineering)** | 成功依赖系统化的提示工程、上下文窗口管理、结构化指令传达细微需求 |
| 7 | **专业知识的再分配 (Redistribution of Expertise)** | Vibe Coding 不消除编程专业知识——它将其重新分配到上下文管理、快速代码评估、以及何时用 AI vs 手动编码的判断上 |

## 五种开发模型

### 模型 1：无约束自动化模型 (Unconstrained Automation)
- **特征**：高信任度、最少人为干预、"Accept All" 模式
- **适用**：原型、实验、探索性项目
- **风险**：代码可能完全不可维护
- **操作**：给定目标 → AI 生成 → 运行 → 报错就粘贴 → 循环直到工作

### 模型 2：迭代对话协作模型 (Iterative Conversational Collaboration)
- **特征**：与 AI 持续对话，逐步细化
- **适用**：功能开发、重构、调试
- **节奏**：描述意图 → 审查结果摘要（不看细节）→ 反馈调整 → 重复
- **关键**：每次对话只聚焦一个变更方向

### 模型 3：计划驱动模型 (Planning-Driven)
- **特征**：先由人提供架构/设计计划，AI 在约束内实现
- **适用**：复杂系统、多模块项目
- **流程**：
  1. 人工定义：模块边界、数据流、接口契约
  2. AI 逐个模块生成
  3. 集成测试验证
- **优势**：可维护性远高于模型 1

### 模型 4：测试驱动模型 (Test-Driven)
- **特征**：先写测试（人或 AI），AI 生成代码通过测试
- **适用**：有明确输入/输出的功能
- **流程**：
  1. 描述预期行为 → AI 生成测试用例
  2. 审查测试（确保测试正确）
  3. AI 生成实现代码 → 运行测试 → 红-绿循环
- **优势**：测试即文档，重构有安全网

### 模型 5：上下文增强模型 (Context-Enhanced)
- **特征**：大量提供项目上下文、文档、约束来引导 AI
- **适用**：大型项目、遗留系统、需要遵循特定规范的场景
- **操作**：
  - 维护 CLAUDE.md / 项目上下文文件
  - 粘贴相关的现有代码片段
  - 明确技术栈、编码规范、架构约束
  - 每次对话前刷新上下文

## 上下文工程策略

### 结构化提示模板

```
[角色] 你是一个资深 [技术栈] 开发者
[上下文] 项目位于 [路径]，使用 [框架/库]
[约束] 遵循 [编码规范]，保持与现有代码风格一致
[任务] 实现 [功能描述]
[输出] 生成 [文件路径] 的完整代码
[验证] 代码应通过 [测试命令]
```

### 上下文窗口管理

1. **分层提供信息**：先给概要，需要时再展开细节
2. **相关性剪枝**：移除无关的历史对话
3. **关键文件先行**：优先粘贴核心接口/配置文件
4. **错误上下文**：粘贴完整错误栈 + 相关代码，不截断
5. **重置策略**：当 AI 开始"兜圈子"时，开启新对话

### 上下文刷新检查清单

- [ ] 项目技术栈声明
- [ ] 当前分支/工作目录
- [ ] 最近 3 个 commit 摘要
- [ ] 相关文件内容（不超过上下文窗口的 30%）
- [ ] 明确的输出格式要求

## 迭代协作模式

### 标准 Vibe 循环

```
1. 说 (Say)：用自然语言描述你想要什么
2. 生成 (Generate)：AI 产生代码
3. 看 (Glance)：快速扫描——结构对么？文件对么？
4. 跑 (Run)：执行并观察行为
5. 反馈 (Feedback)：
   - 成功了 → 说下一个需求
   - 报错了 → 粘贴错误，不加解释
   - 行为不对 → "不对，我要的是 X，现在却是 Y"
6. 接受 (Accept)：Accept All，继续
```

### 反模式

| 反模式 | 正确做法 |
|--------|---------|
| 逐行审查所有 AI 代码 | 信任但验证——看结构和行为 |
| 用代码术语描述需求 | 用自然语言描述想要的结果 |
| 一次提太多需求 | 一次一个变更方向 |
| 在 AI 卡住时反复尝试 | 开新对话，重新提供上下文 |
| 忽略错误信息 | 完整粘贴错误栈 |
| 手动修改大量 AI 代码 | 描述不满意之处，让 AI 重做 |

### 逃逸舱口

当 Vibe Coding 失效时（AI 反复生成错误代码、bug 越修越多）：

1. **回退**：`git checkout -- .` 回到干净状态
2. **缩小**：把需求拆成更小的步骤
3. **切换模型**：从计划驱动切换到测试驱动
4. **人工介入**：手动写关键逻辑，其余交给 AI
5. **重启**：新对话 + 精简上下文

## 质量与风险管理

### 适合 Vibe Coding

- ✅ 原型和 MVP
- ✅ 个人工具/脚本
- ✅ 探索性项目
- ✅ UI 组件和布局
- ✅ 学习和实验
- ✅ 非关键路径功能
- ✅ 有测试覆盖的模块

### 不适合 Vibe Coding

- ❌ 安全关键系统（金融交易、医疗设备）
- ❌ 生产环境核心基础设施
- ❌ 需要深度领域知识的专业系统
- ❌ 加密/认证/授权逻辑（除非有专业审查）
- ❌ 大规模数据迁移
- ❌ 初学者的编程学习

### 安全护栏

1. **永远不信任认证/加密代码**：AI 生成的安全代码必须专业审查
2. **敏感数据隔离**：不在提示中粘贴密钥、令牌、用户数据
3. **依赖审查**：`npm install`/`pip install` 前检查新增依赖
4. **Git 安全网**：频繁 commit，方便回退
5. **环境隔离**：使用 Docker/venv 隔离 AI 生成的代码
6. **端口/网络审查**：检查代码是否打开了意外的网络连接

### 质量信号

| 好信号 | 坏信号 |
|--------|--------|
| 代码结构清晰，文件拆分合理 | 单个文件超过 500 行 |
| 函数短小，命名有意义 | 变量名是 a/b/c/foo/bar |
| 错误处理覆盖了主要路径 | 空的 catch 块 |
| 有基本的输入验证 | 没有任何验证 |
| 依赖数量合理 | 引入了不必要的重型框架 |

## Vibe Engineering —— 进阶实践

由 **Simon Willison** (2025.10) 提出，将 Vibe Coding 升级为生产级实践：

1. **计划先行**：生成代码前先写计划文档
2. **测试覆盖**：每个功能都有测试
3. **文档同步**：AI 生成代码的同时生成文档
4. **代码审查**：AI 生成的代码仍需人工审查关键部分
5. **增量交付**：小步提交，频繁部署
6. **监控就绪**：代码内置日志和健康检查

### Vibe Coding vs Vibe Engineering 选择指南

```
需要快速原型？            → Vibe Coding 模型 1
需要可维护的功能？         → Vibe Coding 模型 3/4
需要生产级质量？           → Vibe Engineering
需要通过合规审计？         → 传统软件开发 + AI 辅助
```
